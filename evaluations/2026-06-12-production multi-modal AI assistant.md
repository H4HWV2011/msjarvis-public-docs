## II. System Architecture Overview

To understand why the failures described in this session occurred, it is necessary to first describe what the Ms. Jarvis / Ms. Allis system is and how it is organized.

### II.A. What the System Is

The system is a production multi-modal AI assistant deployed under the persona name **Ms. Egeria Allis**, presented to users through the web application at `egeria.mountainshares.us`. The system is purpose-built to serve the **MountainShares community ecosystem** in Fayette County and surrounding areas of West Virginia — a region characterized by Appalachian cultural heritage, post-coal economic transition, and active civic data infrastructure projects. The system's design philosophy centers on *data sovereignty*, meaning that the AI's knowledge base is grounded in locally-held, community-specific data rather than generic internet knowledge.

The persona Ms. Egeria Allis is the user-facing identity of a much deeper technical architecture. The underlying system is called **Ms. Jarvis** internally, a reference inherited from the project's early design phase. The public-facing product at `egeria.mountainshares.us` presents the same AI under the name Egeria Allis, who is described as a community intelligence assistant rooted in Appalachian traditions.

### II.B. Container Architecture

The system is deployed as a Docker Compose stack on a single host. The relevant containers participating in this session's work include:

- **jarvis-brain-orchestrator** — the primary orchestration layer that coordinates multi-LLM consensus computation and routes requests to sub-mind models
- **jarvis-wv-entangled-gateway** — a West Virginia–specific routing gateway that handles regional query augmentation
- **jarvis-unified-gateway** — the external-facing entry point that receives user messages, routes them through the appropriate internal pipeline (greeting gate, standard pipeline, or Egeria passthrough), and assembles final responses
- **jarvis-local-resources-db** — a PostgreSQL instance with PostGIS extensions, hosting both the `msjarvisgis` geospatial database and the `local_resources` equity and community data database
- **jarvis-gis-rag** — a retrieval-augmented generation service specifically designed to query geospatial and infrastructure data from the WV GIS Technical Center
- **jarvis-rag-server** — the primary RAG server that serves semantic similarity searches from ChromaDB vector collections
- **jarvis-neurobiological-master** — a service representing a neurological simulation or cognition layer within the system's larger model of AI consciousness
- **jarvis-consciousness-bridge** — a service that bridges the neurobiological master to the broader pipeline
- **jarvis-hilbert-state** — a Hilbert-space arbitration service intended to mediate divergent responses from the multi-LLM ensemble when consensus is not reached
- **jarvis-auth-api** — the authentication and user data service, responsible for serving timesheet / equity metrics session data (EMS) to authenticated users

### II.C. The Multi-LLM Ensemble Architecture

A core design feature of this system is its use of multiple language models simultaneously — referred to internally as "minds" — to generate and arbitrate responses. During the session, the following models were observed participating: LLaMA 3.1, Mistral, Qwen2, Qwen2.5 Mini, Phi3 Mini, Zephyr, Starling-LM, Vicuna-7B, Orca-Mini, Dolphin-Phi, and Gemma. Each model generates a candidate response, and the system then computes a `divergence_score` and `divergence_flag` to measure inter-model disagreement. High divergence — when the models disagree substantially — is supposed to trigger behavioral modifications to the response: either a disclaimer, Hilbert-state arbitration, or both.

The behavioral defects documented in the June 7 engineering report (Defects B1, B2, B3, and B3b) arose precisely from this ensemble: the multi-model architecture was producing outputs that leaked internal attribution language ("Mind 3 asserts...", "As some minds pointed out..."), universally prepended introspective preambles, and universally appended Appalachian community framing — regardless of whether the query was topically relevant to any of those elements.

### II.D. The GBIM System

A specific named subsystem central to this session is the **GBIM**, or **Greeter Brain Integration Module**. The GBIM is not a conversational system in the ordinary sense. Its purpose is to serve as the *knowledge retrieval backbone* for the greeting gate — the logic that intercepts simple, social, or greetings-category user messages before they are routed into the full expensive multi-LLM pipeline. In earlier versions of the system, the greeting gate returned a hardcoded early response when it detected greetings, hello messages, or the phrase "mamma carrie." The GBIM transforms this from a simple bypass into an *informed* bypass: when a greeting is detected, the GBIM queries the ChromaDB vector database to retrieve relevant knowledge snippets and augments the response with actual grounded content, while still avoiding the 11-mind ensemble overhead.

The three ChromaDB collections that `gbim_msjarvis.py` was rewritten to sweep in this session are:

- `autonomous_learner` — contains session-derived learning content accumulated by the system's autonomous learner process
- `appalachian_cultural_intelligence` — contains community knowledge documents specific to Appalachian history, traditions, and regional context
- `conversation_gbim_public` — contains prior conversational context indexed for public-facing retrieval

## V. Port 8018 Contention — Root Cause Discovery and Resolution

### V.A. The Recurring Crash Loop

Despite the June 11 report noting that the port 8018 crash-loop had been resolved, live evidence showed it had recurred. The `jarvis-chat.service` was again crash-looping with `[Errno 98] error while attempting to bind on address ('0.0.0.0', 8018): address already in use`. The `ss`, `lsof`, and Docker process listing commands revealed:

```
python3 1749221 cakidd  13u  IPv4  4616313  TCP *:8018 (LISTEN)
jarvis-neurobiological-master    8018/tcp
jarvis-consciousness-bridge      127.0.0.1:8020->8018/tcp
```

### V.B. The Root Cause

The `jarvis-neurobiological-master` container was configured in `docker-compose.yml` with a bare `"8018"` port declaration (line 859) rather than a host-scoped or expose-only declaration. In Docker Compose, a bare port string with no colon-separated host binding causes Docker to bind the port to `0.0.0.0` on the host — making the container invisible to `docker ps --ports` formatted output while still occupying the host port. This is a well-known Docker networking subtlety that has frequently been misdiagnosed in this system as a Python process issue.

The `jarvis-consciousness-bridge` container had a correct host-scoped binding (`127.0.0.1:8020:8018`), meaning it bound to host port 8020 and forwarded to its internal port 8018. It was not itself blocking port 8018 on the host but was stopped alongside `jarvis-neurobiological-master` as a precaution.

### V.C. The Resolution

Stopping both containers freed port 8018. The resolution commands were:

```bash
docker stop jarvis-neurobiological-master jarvis-consciousness-bridge
kill -9 1749221
sudo systemctl stop jarvis-chat && sleep 2 && sudo systemctl start jarvis-chat
curl -s http://localhost:8018/health | jq .
```

The `jarvis-chat.service` responded immediately with:

```json
{"status": "ok", "service": "ms_jarvis_production_chat", "version": "2.1.0-patched"}
```

A systemd drop-in configuration was also created at `/etc/systemd/system/jarvis-chat.service.d/port-clear.conf` with an `ExecStartPre` directive that runs `fuser -k 8018/tcp` before every service start, providing a permanent structural guard against future recurrence of this class of failure.

### V.D. The Deeper Implication for Port Architecture

The `docker-compose.yml` grep revealed that `NEURO_MASTER_URL` was set to `http://jarvis-neurobiological-master:8018`, and `PSYCHOLOGY_SERVICES_URL` pointed to `jarvis-psychology-services:8019`. This means that `jarvis-neurobiological-master` is an *internal service consumer* — other services call into it by name over the Docker network. It does not need to be reachable on the host network at all. The correct permanent fix is to change the `ports:` declaration to an `expose:` declaration in `docker-compose.yml`:

```yaml
# BEFORE (squatting host 8018):
ports:
  - "8018"

# AFTER (internal only — no host binding):
expose:
  - "8018"
```

Then redeploy:

```bash
docker compose up -d --force-recreate jarvis-neurobiological-master
sudo ss -tlnp | grep 8018   # should now show NOTHING
```

## VI. The Behavioral Defects (B1, B2, B3, B3b)

### VI.A. What These Defects Are

The behavioral defect series designated B1 through B3b describes four distinct categories of inappropriate AI response behavior observed in the live system:

- **B1** — A stereotyped "genuine interiority" preamble was being prepended to all responses, including purely mathematical queries, regardless of whether the query had any introspective or consciousness-related content
- **B2** — Sub-mind attribution language ("Mind 3 asserts...", "As some minds pointed out...") was leaking into the final response prose, exposing the internal multi-LLM ensemble mechanics to the user
- **B3** — An Appalachian/community frame closing was being appended universally, including to responses about mathematics, unrelated scientific topics, and farewell messages
- **B3b** — A farewell phrase from the user ("Good night, Ms. Allis") was causing the system to synthesize a lengthy response recycling prior session context about Appalachian music, data sovereignty, and community knowledge, instead of returning a clean goodbye

These defects collectively undermine the coherence and professionalism of the system's user-facing responses. For a system whose stated identity is as a trusted community AI assistant, the leakage of internal model-naming conventions and the inappropriate injection of ideological framing into every response — including trivial arithmetic queries — represents a fundamental failure of the system's behavioral contract with its users.

### VI.B. Discovery of the Wrong File Path

The June 7 engineering report had designed patches for all four defects, but those patches had been applied to a file that did not exist at the applied path:

```
/opt/msjarvis-rebuild/ms_jarvis_production_chat.py   ← does NOT exist
```

The actual live file is at:

```
/opt/msjarvis-rebuild/services/ms_jarvis_production_chat.py
```

As confirmed by the systemd service unit:

```ini
[Service]
User=cakidd
WorkingDirectory=/opt/msjarvis-rebuild
ExecStart=/opt/msjarvis-rebuild/.venv/bin/python3 services/ms_jarvis_production_chat.py
```

This is one of the most consequential errors in the engineering history of this system: every behavioral fix designed over multiple sessions had been written to a path that does not exist, while the actual running service continued to execute the original, unpatched code. The crash-loop behavior had previously masked this fact, because the service was restarting so rapidly that it was never stable long enough for the behavioral defects to be clearly observed. Once port 8018 was cleared and the service ran stably, the defects became immediately and visibly apparent.

### VI.C. The Patch and Its Verification

The divergence and RAG banner injection block was located precisely using line-number-confirmed grep output at lines 353–367 of `services/ms_jarvis_production_chat.py`. The block unconditionally prepended the epistemic disclaimer and the RAG verification note to every response whenever `divergence_flag` was true.

The patch introduced four runtime classifiers computed from `req.message.lower()` at the point of divergence handling:

- `_is_farewell_div` — detects farewell phrases ("good night", "goodbye", "take care", etc.)
- `_is_trivial_div` — detects short messages (12 words or fewer, a heuristic for simple social queries)
- `_is_locale_div` — detects locale-relevant content (Fayette, MountainShares, Appalachia, West Virginia, etc.)

The epistemic disclaimer is suppressed for farewell and trivial messages. The RAG verification note is now conditional on both `rag_used` and `_is_locale_div` being true, and on `_is_farewell_div` being false — meaning it only appears when the response is both grounded in local documents *and* relevant to the community domain.

Despite confirmed patch success (the verification output showed `_is_farewell_div: True`, `_is_trivial_div: True`, `old unconditional Note: gone: True`), the banners continued to appear in subsequent test responses. Investigation suggested the live response pipeline was not reaching the patched block via the expected code path, or that a second divergence_flag evaluation block exists elsewhere in the file. This remains an open investigation item requiring further inspection of the full divergence handling logic.

## VII. The wv_layer_registry Schema Incident

### VII.A. What This Table Is

The `wv_layer_registry` table in the `msjarvisgis` PostgreSQL database is a catalog table that records metadata about each GIS layer loaded into the system from the West Virginia GIS Technical Center (WVGISTC). It enables the `jarvis-local-resources` service to enumerate available geospatial layers, understand their source agencies, coordinate reference systems (SRID), and geometry column names, which are required for dynamic spatial query construction.

### VII.B. The Schema Conflict

The original table had the following schema:

```sql
id            SERIAL PRIMARY KEY,
layer_name    TEXT NOT NULL UNIQUE,
source_agency TEXT,
source_date   DATE,
geometry_type TEXT,
srid          INTEGER,
row_count     INTEGER,
description   TEXT,
loaded_at     TIMESTAMPTZ DEFAULT now(),
geom_column   TEXT DEFAULT 'geom'
```

An attempt was made during the session to INSERT records with a `display_name` column that did not exist in this schema, producing the PostgreSQL error:

```
ERROR:  column "display_name" of relation "wv_layer_registry" does not exist
```

### VII.C. The Drop-and-Recreate Sequence

In diagnosing the error, the table was dropped and recreated — but with a *different* schema that omitted `id`, `source_agency`, `source_date`, `srid`, `row_count`, and `geom_column`. The 12-row INSERT succeeded. However, in a subsequent command block, a second `DROP TABLE` command executed as part of the next diagnostic block, and the `CREATE TABLE` that was supposed to follow it contained a shell-level syntax error that caused the heredoc to abort mid-string. The table was dropped a second time and never recreated.

The result is that `wv_layer_registry` is completely absent from the database at session close. The correct remediation is to recreate the table using the original schema and repopulate it from the ingest scripts rather than hand-written INSERT statements:

```sql
CREATE TABLE IF NOT EXISTS wv_layer_registry (
  id            SERIAL PRIMARY KEY,
  layer_name    TEXT NOT NULL UNIQUE,
  source_agency TEXT,
  source_date   DATE,
  geometry_type TEXT,
  srid          INTEGER,
  row_count     INTEGER,
  description   TEXT,
  loaded_at     TIMESTAMPTZ DEFAULT now(),
  geom_column   TEXT DEFAULT 'geom'
);
```

## VIII. The GBIM Rewrite and Deployment (gbim_msjarvis.py)

### VIII.A. Context and Purpose

The session's final major engineering action was the rewrite and deployment of `gbim_msjarvis.py` and its companion files `gbim_orchestrator.py` and `gbim_chroma.py`. The deployment targeted all three primary service containers: `jarvis-brain-orchestrator`, `jarvis-wv-entangled-gateway`, and `jarvis-unified-gateway`.

The GBIM system serves as the knowledge retrieval component of the greeting gate pathway in the unified gateway. When a user message is classified as a greeting or simple social interaction, it is intercepted by the greeting gate *before* being sent to the full multi-LLM ensemble. The GBIM's job is to ensure that this early-return pathway is not informationally empty — rather than returning a generic greeting, it queries the ChromaDB vector database to retrieve relevant knowledge, enabling grounded, contextually aware greetings.

The deployment was executed as follows:

```bash
for container in jarvis-brain-orchestrator jarvis-wv-entangled-gateway jarvis-unified-gateway; do
    docker exec $container find /app -name "gbim_msjarvis.py" 2>/dev/null && \
        docker cp /opt/msjarvis-rebuild/services/gbim_msjarvis.py \
            $container:/app/services/gbim_msjarvis.py && \
        echo "Deployed"

    docker exec $container find /app -name "gbim_orchestrator.py" 2>/dev/null && \
        docker cp /opt/msjarvis-rebuild/gbim_orchestrator.py \
            $container:/app/gbim_orchestrator.py && \
        echo "Deployed orchestrator"

    docker exec $container find /app -name "gbim_chroma.py" 2>/dev/null && \
        docker cp /opt/msjarvis-rebuild/gbim_chroma.py \
            $container:/app/gbim_chroma.py && \
        echo "Deployed chroma client"
done

docker restart jarvis-brain-orchestrator jarvis-wv-entangled-gateway jarvis-unified-gateway
```

All three containers confirmed successful deployment of all three files.

### VIII.B. The Three-Collection Sweep Architecture

The rewritten `gbim_msjarvis.py` was designed to sweep three ChromaDB collections:

- **`autonomous_learner`** — populated by the `ms_jarvis_autonomous_learner` process, which continuously ingests and indexes session-derived learning; sweeping this collection ensures the GBIM has access to the most recently learned information
- **`appalachian_cultural_intelligence`** — contains domain-specific documents on Appalachian history, cultural traditions, music, food culture, community organization, and related topics; access to this collection enables the greeting pathway to retrieve culturally grounded context
- **`conversation_gbim_public`** — stores prior conversational content indexed for retrieval through the GBIM pathway, providing continuity across sessions

### VIII.C. The Greeting Gate Fix

The deployment also included a critical modification to the greeting gate logic in `ms_jarvis_unified_gateway.py`. The previous greeting gate contained the phrases "mamma carrie" and "it's mamma" in its bypass pattern list — meaning that any user message containing these strings would trigger the greeting early-return without going through the full pipeline. This was incorrect behavior: "Mother Carrie" and "Mamma Carrie" are references to the **MOTHER_CARRIE_PROTOCOLS**, a named anchor in the system's behavioral configuration representing foundational identity principles for the Ms. Allis persona. Messages referencing these protocols are substantive, not greeting-style, and should route through the full pipeline. The patch removed these strings from the bypass patterns and added a substantive keyword guard to prevent similar misrouting.

### VIII.D. The services_used Field and Pipeline Transparency

A second modification was made to `ms_jarvis_unified_gateway.py`: the Egeria passthrough return path was patched to inject a `services_used` field into `_resp` when one was not already present. The verified output at session close confirmed:

```
"Hi Ms. Jarvis"                               → services_used: ['greeting_gate']
"What is MountainShares and who founded it?"  → services_used: ['egeria', 'gbim_corpus', 'nbb_full_pipeline']
```

This field provides the front-end UI with a consistent, structured indicator of which internal pipeline services participated in generating a given response. Prior to this patch, the Egeria passthrough returned `_resp` directly without populating this field, meaning the UI had no way to distinguish greeting-path responses from full-pipeline responses. Consistency of this field across all code paths is a prerequisite for admin panel observability, A/B testing, and latency attribution.

### VIII.E. The Hilbert-State Timeout Adjustment

The `gbim_orchestrator.py` deployment included a patch that extended the Hilbert-state service timeout from 8 seconds to 15 seconds. This change reflects the reality that the Hilbert-state arbitration process — which involves posting a divergent provisional answer to `jarvis-hilbert-state:8081` and awaiting a mediated response — requires additional processing time under real ensemble load conditions. An 8-second timeout was causing premature fallback to disclaimer-only behavior when the Hilbert service was under load, defeating the purpose of the mediation layer.

## IX. The Autonomous Learner and Embedding Architecture

### IX.A. ChromaDB and Embedding Model Coherence

An important investigative thread in this session concerned the embedding model used by the `jarvis-gis-rag` service. The service's runtime configuration showed `GIS_EMBED_MODEL=all-minilm`, and the Docker container environment confirmed `GIS_EMBED_MODEL=all-minilm`. However, investigation of the ingest scripts raised the concern that the 155,224 GIS vectors may have been ingested using a different embedding model than the one used at query time.

This is a critical structural concern. ChromaDB vector similarity search operates by computing cosine similarity between query embeddings and stored document embeddings. If the query embedding and the stored embeddings were generated by models with different output dimensions — for example, `all-minilm` produces 384-dimensional vectors while `nomic-embed-text` produces 768-dimensional vectors — every search returns either an error or dimensionally incoherent results. The session did not resolve this question because the relevant grep output was cut off before the model lines were displayed. This remains an open investigation item.

### IX.B. The gisgeodb_storage.py Collection Name Note

A structural note of architectural significance: `gisgeodb_storage.py` contains a function that calls `client.get_or_create_collection(name="learner_embeddings")`, which suggests that the autonomous learner process stores its embeddings in a ChromaDB collection called `learner_embeddings`. The GBIM sweep targets `autonomous_learner`, not `learner_embeddings`. It is worth confirming that these collection names are consistent and that the GBIM is sweeping the correct collection for autonomous learner content.

## X. The Hilbert-State Architecture and Its Role in Epistemic Integrity

### X.A. What Hilbert-State Is

The `jarvis-hilbert-state` service at port 8081 represents one of the most architecturally interesting components in this system. Named for the mathematical concept of Hilbert space — a complete inner product space used extensively in quantum mechanics and functional analysis — the service's role is to mediate between divergent probabilistic outputs from the multi-LLM ensemble. When the divergence score exceeds a threshold, instead of simply appending a disclaimer, the system is designed to POST the provisional answer, the divergence score, the mind count, and the user ID to the Hilbert service, which applies a second-stage arbitration process and returns a refined, higher-confidence response.

As of the end of this session, the Hilbert-state service is reachable but its live API contract has not been fully confirmed. The scaffolding in `ms_jarvis_production_chat.py` is present but deactivated, noted in comments as temporarily disabled pending API contract confirmation. The `gbim_orchestrator.py` timeout was extended to 15 seconds to accommodate it. Activating the Hilbert mediation — promoting divergence handling from disclaimer-only to true second-stage arbitration — remains the next high-priority engineering action.

The expected API contract is:

```
POST http://jarvis-hilbert-state:8081/mediate
Body: { query, provisional_answer, divergence_score, mind_count, user_id }
Response: { response, confidence, method }
```

## XII. Methodological Observations

This session exemplifies what the June 11 engineering report identified as the core methodological shift in this system's development trajectory: *from guesswork-based debugging to evidence-driven engineering*. Every diagnostic conclusion in this session was derived from live system output — container process tables, port binding status, exact grep line numbers, systemd journal entries, direct PostgreSQL schema inspection, and live HTTP response bodies from the running API.

The session also demonstrates a recurring pattern in complex multi-container systems: *failure masking*, where a lower-level failure (the port crash-loop) prevents observation of higher-level failures (the behavioral defects). The crash-loop was restarting the service faster than the behavioral defects could be observed in normal operation. Once the port conflict was cleared and the service ran stably, the behavioral issues became immediately and clearly visible, changing the diagnostic priority order entirely.

The question of file path correctness — the discovery that all behavioral patches had been applied to a root-level path that does not exist while the service runs from a subdirectory path — represents the single highest-impact error in the engineering history of this session series. It is recommended that future patching operations include an immediate verification step immediately after writing, before restarting, to confirm the patch landed in the correct location:

```bash
grep -n "<patch_string>" <intended_path>
```

Finally, the architecture of `services_used` as a pipeline transparency field, established at session close, lays the groundwork for operational observability at the UI level. This small structural addition — a consistent dictionary key present on every API response regardless of which internal pathway generated it — enables the admin panel, the front-end, and future debugging sessions to reason about pipeline behavior without requiring log access. It is the correct long-term design pattern for this system's API surface.
